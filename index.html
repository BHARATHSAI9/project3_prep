Why these features required new design

Roller's existing architecture handles content display well (Velocity templates, rendering plugins) but had no built-in support for:

  • User engagement tracking — no concept of user-content relationship beyond authorship
  • Save-time content processing — the existing WeblogEntryPlugin system transforms at render time (SmileysPlugin, ConvertLineBreaksPlugin), but raw content stays untouched in DB. We needed filters that persist the processed content.
  • Event-driven notifications — Roller had no observer/event system; actions directly call managers
  • Configurable reporting — admin pages existed for user/weblog management, not aggregated metrics
  • External API integration — no abstraction for calling third-party services with different interfaces
  • Comment analytics — comments stored and displayed, but no analysis pipeline

Task 1: Stars & Trending 10 marks

Why this change was needed

Roller had no user engagement model beyond authorship and comments. No way to favourite content or surface popular posts. Stars add a user-content affinity model as a first-class JPA entity.

Design decision: single table vs. two tables

OptionProsCons
Single roller_star with target_type enumOne manager, one query set, extensible (add COMMENT starring = add enum value)WHERE clause on type in every query✓ Chosen
Separate tables per typeSimpler per-table queriesDuplicate managers, duplicate trending logic, new table per entity type✗

Why no formal GoF pattern

The core challenge is query optimization, not behavioral variation. The assignment says "do not iterate over individual articles" — the solution is efficient JPQL (GROUP BY / COUNT / LIMIT), not a pattern. Applying Strategy for "different trending algorithms" would be force-fitting a pattern to a fixed requirement.

Key implementation details

  • Trending query: SELECT s.targetId, COUNT(s) as cnt FROM Star s WHERE s.targetType = ...ENTRY GROUP BY s.targetId ORDER BY cnt DESC + setMaxResults(5). Single aggregate query, no per-article iteration.
  • Recency: Starred weblogs sorted by most-recent entry pubTime via JOIN in named query — SQL sorts, not Java.
  • Pagination: JPA offset/limit (page size 20), not in-memory slicing.
  • Duplicate prevention: Unique composite index (userid, target_id, target_type) + manager-level check.

Anticipated TA questions

Q: How do you avoid N+1 for trending?
One aggregate query gets top-N target IDs with counts. Then one batch entity lookup. Total: 2 queries, not N+1.
Q: What about ties in star count?
DB's internal ordering breaks ties. For production: add secondary sort on MAX(s.createdAt). Not required for the assignment.

Task 2: Transforming Feeds Chain of Responsibility 10 marks

Why this change was needed

Roller already had WeblogEntryPlugin (SmileysPlugin, ConvertLineBreaksPlugin, etc.), but those transform at render time — raw content stays untouched in DB. The assignment requires save-time pre-processing where filtered content is persisted. Also, existing plugins have no admin enable/disable UI. We needed a fundamentally different pipeline.

Pattern: CoR vs closest alternatives

PatternHow it would workVerdict
Chain of ResponsibilityIndependent filter objects with apply() + getOrder(), managed in CopyOnWriteArrayList✓ Maps directly to "add/remove steps without affecting remaining process." Runtime enable/disable via admin UI.
DecoratorEach filter wraps the previous, forming nested chain✗ Decorators modify the interface. Our filters operate independently on the same WeblogEntry. Runtime removal = rebuild entire wrapper chain.
PipelineFixed sequence with typed inputs/outputs✗ Implies fixed structure. We need runtime enable/disable without restructuring. Also, Pipeline transforms data between stages; our filters mutate the same object.
StrategyPick one algorithm✗ We need multiple filters to compose in sequence, not pick one-of-many.

Why static rules (not LLMs)

Filters execute in the blog save path. Adding 2-5s of API latency every save would degrade UX. Static rules (regex, word counting, frequency analysis) are deterministic, fast, and have zero external dependencies. The pipeline is extensible — adding an LLM-based filter later = implement ContentFilter.

Integration point (critical to know)

FilterChainManager.getInstance().processEntry(entry) is called at line 231 of JPAWeblogEntryManagerImpl.saveWeblogEntry() — right before strategy.store(entry). Every blog save goes through the pipeline transparently.

Code trace-through

1. User clicks "Save" → Struts2 dispatches → EntryEdit action
2. EntryEdit calls WeblogEntryManager.saveWeblogEntry(entry)
3. JPAWeblogEntryManagerImpl line 231:
   FilterChainManager.getInstance().processEntry(entry)
4. FilterChain.execute(entry) iterates CopyOnWriteArrayList:
   → ProfanityFilter.apply(entry)       [order 10] — regex word-boundary matching
   → TextSummarizerFilter.apply(entry)  [order 20] — truncate to 5000 words
   → TagGeneratorFilter.apply(entry)    [order 30] — top-5 keywords appended
5. Each filter modifies entry.getText() in place
6. strategy.store(entry) → JPA persists the processed content

Anticipated TA questions

Q: Roller has WeblogEntryPlugin — why not use it?
Existing plugins transform at render time; raw content stays in DB. We need save-time processing where filtered content is persisted. Also, existing plugins lack admin enable/disable UI.
Q: What if filters conflict?
Intentional ordering: profanity (10) cleans first, summarizer (20) trims, tag generator (30) extracts keywords from already-cleaned text. Each filter.apply() is wrapped in try-catch — a failing filter doesn't block the save.
Q: Why CopyOnWriteArrayList?
Blog saves (reads) vastly outnumber admin filter toggles (writes). COWAL copies the array only on mutation — optimized for read-heavy patterns.

Task 3: Bug Reports Observer 15 marks

Why this change was needed

Roller had no user feedback mechanism. The assignment requires a notification system that is "modular enough to support addition of new notification channels without any impact to remaining channels." This extensibility requirement drives the pattern choice.

Pattern: Observer vs closest alternatives

PatternHow it would workVerdict
ObserverBugEventPublisher (subject) maintains list of NotificationChannel (observers). Events broadcast to all.✓ Adding Slack = 1 new class + 1 line registration. Zero changes to existing code. Publisher doesn't know channel implementations.
MediatorCentral mediator coordinates between manager and channels✗ Mediator for many-to-many coupling. We have one publisher → N channels. Observer handles this simpler hub-and-spoke naturally.
Event Bus (Guava)Typed events, annotation-based dispatch✗ Would work, but adds external dependency. Observer achieves same decoupling with standard Java. Assignment values pattern understanding.
Direct callsManager calls email/log services directly✗ Violates OCP — adding Slack = modifying the manager. Couples data layer to notification specifics.

Why Mailpit instead of MailerSend

Assignment suggested MailerSend, but it requires account creation, API keys, domain verification. Mailpit is a Docker container that captures all SMTP traffic with a web inbox at localhost:8025. Fully reproducible, zero external dependencies. TA can verify emails without creating accounts.

Why synchronous notifications

Email sending is in the action thread. Each channel call is try-catch wrapped, so a failed email doesn't block other channels. For production: async queue (JMS/Kafka). For this scale: synchronous is simpler and sufficient. Acknowledged as trade-off.

Extensibility proof (show in eval)

Anticipated TA questions

Q: Why singleton instead of Guice DI?
Publisher is a pure in-memory event bus — no JPA needed. Roller's Guice bindings are for managers with JPAPersistenceStrategy. Singleton is consistent with other lightweight utilities (FilterChainManager, TranslationProviderFactory).
Q: How would you actually implement Slack?
Implement NotificationChannel, POST to Slack webhook URL (configured via system property) in each lifecycle method, register at startup. BugReportManager, BugEventPublisher, and all existing channels remain untouched.

Task 5: Translation Adapter + Factory 35 marks (20+15)

Why this change was needed

Roller had no internationalization beyond static locale files. The assignment requires application-side translation that is "easily switchable" between providers. The two APIs have completely different interfaces:

Language codesen-IN, hi-INen, hi
Authapi-subscription-key headerNone (free tier)
Response fieldtranslated_textresponseData.translatedText

Normalizing these incompatible interfaces is the Adapter's job.

Pattern: Adapter + Factory vs closest alternatives

PatternHow it would workVerdict
Adapter + FactoryEach provider adapts its API to TranslationProvider. Factory manages creation/lookup by ID.✓ Adapter isolates API differences. Factory enables switchability — UI dropdown from getAllProviders(). Adding Google Translate = 1 Adapter + 1 register() call.
Strategy aloneProvider selected as runtime strategy✗ Providers aren't just different algorithms — they have different interfaces (endpoints, auth, response formats). The normalization is Adapter's job, not Strategy's.
FacadeSingle facade hides all providers✗ Facade simplifies a complex subsystem. We need providers individually addressable (user picks). Facade hides the choice.
BridgeDecouple abstraction from implementation✗ Bridge separates two independently varying hierarchies. We have one hierarchy and one dimension of variation. Adapter is simpler and more direct.

5B: Two-level caching design

  • L1 (ConcurrentHashMap): Sub-ms lookups. Key = provider:srcLang:tgtLang:contentHash
  • L2 (roller_translation_cache): Persists across restarts. Composite index for fast lookups.
  • Lookup: L1 → L2 (promote to L1 on hit) → API call → store in both
  • Section-level: HTML split by block elements. Each section cached independently. When author edits: only changed sections retranslated.
  • "Significant change": >10% word-level diff after HTML stripping → retranslate. Below → reuse. (e.g., typo fix doesn't retranslate)
  • Cache key includes providerId: Switching Sarvam→MyMemory doesn't serve stale translations.
Known trade-off: String.hashCode() is 32-bit, collision-prone. For production: SHA-256. For project scope: fast and sufficient.
Known trade-off: L1 cache grows unbounded. For production: LRU eviction or TTL. For project scope: resets on restart.

Code trace-through

1. User selects provider + languages → TranslateEntryAction
2. TranslationProviderFactory.getProvider("sarvam") → SarvamTranslationProvider
3. TranslationCache.translateWithCache(text, src, tgt, "sarvam", provider):
   a. Split HTML into sections by <p>, <h1-h6>, <div>
   b. For each section:
      → key = "sarvam:en:hi:{hashCode}"
      → L1 hit? → use cached, next section
      → L1 miss → L2 DB query → hit? → promote to L1, use
      → Both miss → provider.translate(section, "en", "hi")
        → SarvamTranslationProvider maps "en"→"en-IN"
        → POSTs to api.sarvam.ai with api-subscription-key header
        → Extracts "translated_text" from response
      → Store in L1 + L2
   c. Reassemble sections
4. Save TranslationPreference → auto-translate on revisit

Anticipated TA questions

Q: Why MyMemory?
Free tier, no API key, demo-able without external accounts. Both providers support ≥5 languages bidirectionally.
Q: How do you add Google Translate?
Implement TranslationProvider (handle Google's language codes, OAuth2, response JSON). Call factory.register(). Auto-appears in UI dropdown. Cache works unchanged — key includes providerId.
Q: What's a "significant change"?
Strip HTML, normalize whitespace, compare word-by-word. >10% differ → significant. The hasSignificantChange() method in TranslationCache implements this. 10% balances freshness vs. API cost.

Task 6: Community Pulse Strategy 15 marks

Why this change was needed

Comments were stored and displayed but never analyzed. Authors read every comment to gauge sentiment. The assignment requires "multiple insight-generation methods depending on available resources" — explicitly a Strategy scenario. The 6A/6B split mandates lightweight classical methods AND AI-powered analysis coexisting.

Pattern: Strategy vs closest alternatives

PatternHow it would workVerdict
Strategy3 strategies behind InsightStrategy, all returning PulseInsight. Selected at runtime.✓ Classic Strategy. User/system picks the algorithm. JSP renders identically regardless of which strategy ran.
Template MethodAbstract class with shared steps, subclasses override specifics✗ Our strategies share almost nothing — LLM strategy samples comments differently than keyword strategy. Template Method forces a shared skeleton that doesn't exist.
Chain of ResponsibilityStack analyses in sequence✗ We want to pick one based on resources, not run all in sequence.

3 strategies (exceeds required 2)

  1. KeywordFrequencyStrategy (6A) — word counting + stop-word filtering. Indicators: activity level, top-10 keywords, representative comments.
  2. SentimentHeuristicStrategy (6A) — predefined positive/negative/question word lists. Classifies comments, produces sentiment distribution.
  3. LlmInsightStrategy (6B) — GroqCloud Llama 3.1-8b-instant. Selective LLM usage: keywords extracted locally, LLM only for theme synthesis + recap. Max 20 comments sampled. 10s timeout.

"Don't use a nuke to kill a fly"

LLM does NOT extract keywords — that's trivial statistically. LLM used only for what statistics can't do: synthesizing themes into readable sentences and generating natural-language recaps.

Graceful fallback

isAvailable() checks for groq.api.key. Missing → auto-falls back to KeywordFrequencyStrategy + shows banner. Feature works with zero API keys.

Anticipated TA questions

Q: 20 comment cap — what about 500+ comments?
Evenly sample: step = total/20, pick every step-th comment for representative distribution. Each truncated to 300 chars. Keyword extraction runs on ALL comments locally. Only the LLM receives the sample.
Q: Manual JSON parsing — why not Jackson?
Avoid adding a dependency. OpenAI-compatible response is predictable. If parsing fails, graceful fallback to keyword-based analysis.
Q: How does 6A vs 6B map?
6A = "classical, inexpensive methods" → KeywordFrequency + SentimentHeuristic. 6B = "heavier AI" → LlmInsight. All unified under InsightStrategy; user picks from dropdown.

Pattern-level

Q: Which pattern are you most proud of?
Adapter + Factory for translation. It solved a real problem — two APIs with incompatible interfaces. The Adapter normalizes them; the Factory makes them switchable. Cache integrates transparently. Extensibility is tangible: third provider = zero existing code changes.
Q: If you could redo one pattern?
Dashboard Builder. 11 classes for 2 views is heavy. Simpler enum-based metric selection + factory might achieve the same separation. But the assignment's explicit requirement practically demands Builder.
Q: How do patterns interact across tasks?
Three integrations: (1) MostStarredBlogProvider → StarManager (Task 1), (2) BugReportSummaryProvider → BugReportManager (Task 3), (3) Cache key includes providerId → works with any Adapter. Patterns compose, not isolated.

Architecture-level

Q: How did you integrate with Roller's existing architecture?
3 new Guice bindings in JPAWebloggerModule. Exposed through Weblogger interface. ORM XML mappings. Struts2 actions with existing interceptor stacks. No new frameworks.
Q: Why singletons (FilterChainManager, BugEventPublisher, etc.) instead of Guice?
These are lightweight in-memory utilities — no JPA needed. Roller's Guice bindings are for managers using JPAPersistenceStrategy. Singletons are consistent with each other. Refactoring to Guice would be minimal if needed.
Q: What would break if requirement X changed?
"Per-weblog filters": FilterChainManager storage changes (Map<weblogId, FilterChain>). ContentFilter interface unchanged.
"Translate comments too": New action sibling. Adapter, Factory, Cache all unchanged.
"Bug priority escalation": New NotificationChannel. Existing channels untouched.
Answer pattern: "Only class X changes" = proof of modularity.
